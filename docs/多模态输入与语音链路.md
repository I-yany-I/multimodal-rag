# 多模态输入与语音链路说明

## 1. Qwen2-VL 在本项目里实际吃的是什么？

当前生成端 **Qwen2-VL** 通过 `processor` 接收的是：

- **文本**：用户问题（以及内部的 query 改写提示等）；
- **图像**：检索到的若干张 **PIL 图像**（来自图库路径读入）。

也就是说：**原生送进 VLM 的是「文本 + 图像」两类张量**，与常见「VLM = 视觉 + 语言」一致。

**音频**并没有作为第三种连续信号送进 Qwen2-VL。语音路径是：

```text
麦克风/音频文件 → faster-whisper（ASR）→ 纯文本 query → 与「文本问答」完全相同的 CLIP 检索 + VLM 生成
```

因此，若面试官问「是不是三种模态同时进 VLM？」更严谨的回答是：

- **不是**「音频波形 + 图像像素 + 文本 token」三流合一进同一个声学编码器；
- **是**「音频先被 **模态转换** 成文本，再与图像一起做多模态推理」，这是工业界非常常见的 **级联（cascade）** 设计，便于复用现有文本检索与 VLM。

若要做 **真·三模态单模型**（声纹/环境声与画面联合推理），需要带 **音频塔** 的统一多模态大模型或单独音频检索（如 CLAP）+ 融合层，复杂度和数据管线都会明显上升。

## 2. 语音实现算不算「太简略」？

**简略是有意取舍**，不是偷懒：

| 做法 | 优点 | 缺点 |
|------|------|------|
| **当前：ASR → 文本 RAG** | 工程清晰、易评测、显存好控、和文本 Tab 逻辑一致 | 丢韵律/说话人/非语义声音信息 |
| 深化方向（可作为后续工作） | 时间戳段落、**Whisper word-level** 对齐、**说话人分离**、**CLAP 音频向量库** 与图像双路检索 | 开发量、评测与延迟都上去 |

简历/面试里可以一句话：**「语音走级联 ASR，优先保证产品路径闭环；深度音频语义留给有业务需求再迭代。」**

## 3. 个人图库照片来源（合规）

- **正式用途**：只应用 **本人拥有使用权** 的照片（手机导出、相机拷贝）。
- **演示/填空**：可用本仓库提供的 **`scripts/seed_demo_personal_images.py`**，从 **固定 ID 的占位图服务** 拉取示例图（非真实隐私生活照），仅用于跑通索引与 UI。
- **不建议、也不会实现**：从搜索引擎/社交平台 **批量爬「个人生活照」**——版权与隐私风险极高，不适合放进简历项目。
